# AGENTS.md -- AI Agent Guide for action-omeka-s-playground-pr-preview

## Origin and Attribution

This project is a sibling of [ateeducacion/action-moodle-playground-pr-preview](https://github.com/ateeducacion/action-moodle-playground-pr-preview): both are GitHub Actions that post a "preview this PR in a browser-based runtime" link on pull requests. This one targets **Omeka S** instead of Moodle.

- **This repo:** `ateeducacion/action-omeka-s-playground-pr-preview`
- **Companion project:** `ateeducacion/omeka-s-playground` -- the browser-based Omeka S runtime (PHP-WASM) that this action links to
- **Sibling action:** `ateeducacion/action-moodle-playground-pr-preview` -- same idea for Moodle plugins

Note: while the two actions share a concept and some behaviors (sticky comment/description management, base64url blueprint encoding, URL-length warning), their code is **not** shared. This repo does **not** have Moodle's `blueprint-file`, `proxy-url`, or core-overlay features -- do not assume they exist here.

## What This Project Does

This is a **GitHub Action** that automatically posts (or updates) a "Preview in Omeka S Playground" link on a pull request. When a contributor opens a PR on an Omeka S module or theme repository, this action generates a link that opens the PR's addon ZIP directly inside an ephemeral, in-browser Omeka S instance powered by Omeka S Playground.

The action does **not** build, test, or deploy anything. It only:

1. Reads PR metadata and the addon ZIP URL (passed in via inputs).
2. Synthesizes an Omeka S Playground blueprint JSON from the discrete action inputs.
3. Encodes the blueprint as base64url and builds a Playground URL.
4. Posts the link to the PR (either as a sticky comment or in the PR description).

## How to Build

```bash
npm install
npm run build    # runs: esbuild index.js --bundle --platform=node --target=node20 --format=cjs --outfile=dist/index.cjs
```

The build bundles `index.js` and its `node_modules` dependencies into a single CommonJS file `dist/index.cjs`. `action.yml` runs `dist/index.cjs` directly (`runs.using: node20`, `runs.main: dist/index.cjs`). **Always rebuild and commit `dist/index.cjs` after editing `index.js` or `lib.js`** -- the CI `verify-dist` job fails the build if the committed bundle is out of sync.

## Source Code Architecture

The code is split into two plain-JavaScript ESM files (CommonJS is only the bundle format; the sources use `import`/`export`). There is no TypeScript.

- **`index.js`** -- the runtime glue. A single async `run()` function reads every input via `core.getInput`, resolves the PR context (from the `pull_request` event payload, or via the GitHub API when `pr-number` is supplied for `workflow_run` triggers), calls the pure helpers in `lib.js` to build the blueprint and preview URL, then publishes the result to the PR via Octokit. All side effects (Octokit calls, `core.setOutput`, `core.warning`, `core.setFailed`) live here.
- **`lib.js`** -- pure, side-effect-free logic that is unit-tested in isolation: blueprint construction, deep-merge of the `blueprint-json` override, addon normalization/dedup, base64url encoding, URL building, the URL-length safety check, comment/description body templating, and the sticky-block regex helpers.

### Execution Flow (`index.js` `run()`)

```
1. Read inputs (github-token, zip-url, addon-*, title/description/author,
   playground-url, image-url, comment-marker, mode, extra-text,
   restore-button-if-removed, pr-number, and all blueprint inputs:
   extra-modules/extra-plugins, extra-themes, users-json, item-sets-json,
   items-json, site-json, landing-page, debug-enabled, site-*, login-*,
   blueprint-json).
2. Validate mode (comment | append-to-description); setFailed on invalid.
3. Resolve PR context:
   - From event payload (pull_request trigger), OR
   - From GitHub API if pr-number input is provided (workflow_run trigger).
4. buildBlueprint(...) -> synthesize the blueprint object from inputs.
5. JSON.stringify the blueprint, buildPreviewUrl(...) -> base64url ?blueprint-data= URL.
6. If previewUrlExceedsLimit(previewUrl) -> core.warning(...) (advisory only, never fails).
7. setOutput preview-url and mode.
8. Publish:
   - MODE_APPEND: update PR body with the managed :start/:end block.
   - MODE_COMMENT (default): first strip any leftover description block, then
     create or update the sticky bot comment identified by the hidden marker.
```

### Blueprint Format

`buildBlueprint()` always emits `$schema` and `meta`, plus a primary addon derived from `zip-url` + `addon-type`:

- `addon-type: module` (default) -> a `modules` entry `{ name, state, source: { type: "url", url: <zip-url> } }` where `state` is `install` or `activate` (default `activate`).
- `addon-type: theme` -> a `themes` entry `{ name, source: { type: "url", url: <zip-url> } }` (no `state`).

The addon `name` comes from `addon-name`, or is inferred from a GitHub archive URL when possible; if neither yields a name, the action throws.

Optional top-level sections are added only when their inputs are present: `modules`/`themes` (extended by `extra-modules`/`extra-themes`), `users`, `itemSets`, `items`, `site`, `landingPage`, `debug` (`{ enabled }`), `siteOptions` (`{ title, locale, timezone }`), and `login` (`{ email, password }`). `modules` and `themes` are deduplicated by addon name (string entries and `{ name }` objects). Finally, the `blueprint-json` input is deep-merged **last** as an override layer, after which `modules`/`themes` are deduplicated again.

Minimal generated blueprint:

```json
{
  "$schema": "https://ateeducacion.github.io/omeka-s-playground/assets/blueprints/blueprint-schema.json",
  "meta": {
    "title": "PR Preview",
    "author": "ateeducacion",
    "description": "Preview this PR in Omeka S Playground"
  },
  "modules": [
    {
      "name": "MyModule",
      "state": "activate",
      "source": {
        "type": "url",
        "url": "https://github.com/OWNER/REPO/archive/refs/heads/BRANCH.zip"
      }
    }
  ]
}
```

Addon `source` also supports non-URL types (e.g. `{ "type": "omeka.org", "slug": "numeric-data-types" }`) when passed through `extra-modules`/`extra-themes` or `blueprint-json`.

### Preview URL Encoding

Omeka S Playground accepts the blueprint inline via a single query parameter:

```text
https://ateeducacion.github.io/omeka-s-playground/?blueprint-data={base64url-encoded-json}
```

`buildPreviewUrl()` ensures the base URL ends with `/` before appending `?blueprint-data=`. There is **no** remote-URL / proxy alternative in this repo -- the blueprint is always inlined.

Because the whole blueprint is inlined, large blueprints produce long URLs. After building the URL, `index.js` calls `previewUrlExceedsLimit(previewUrl)`; if the URL exceeds `MAX_SAFE_PREVIEW_URL` (8000 chars) it emits a `core.warning` suggesting the user trim `extra-modules`/`extra-themes`/`items-json`/`site-json` or split into a smaller blueprint. This is **advisory only** -- it never fails the action (matching the Moodle sibling's warn-don't-block behavior).

### Publish Modes

The action supports two publish modes via the `mode` input; `comment` is the default.

**`comment` (default).** Searches PR comments (lazy pagination, 100 per page, stops at the first match) for one containing the hidden `<!-- {comment-marker} -->` marker. Updates it if found, otherwise creates a new sticky comment. Before doing so it **cross-cleans**: if a managed block from `append-to-description` mode is still in the PR body (e.g. the user switched modes), `removeDescriptionBlock` strips it so the preview lives in exactly one place. Sets the `comment-id` output.

**`append-to-description`.** Wraps the preview in HTML comment markers inside the PR body:

```html
<!-- omeka-s-playground-preview:start -->
...preview button + link...
<!-- omeka-s-playground-preview:end -->
```

`computeNextDescriptionBody` finds and updates this block on subsequent runs, and respects user modifications:

- If the user replaced the content between the markers with their own text (no `<a ...>` containing "playground"), it is treated as a **placeholder** and the body is left untouched (`null`).
- If the markers are absent and `restore-button-if-removed` is `false`, the body is left untouched; if `true` (default), the block is appended to the end.

Sets the `rendered-description` output.

## CI Workflows

### ci.yml

Runs on push to `main` and on PRs. Two jobs:

- **test** -- `npm ci` then `npm test`.
- **verify-dist** -- `npm ci`, `npm run build`, then fails if `git status --porcelain dist/` is non-empty (i.e. the committed bundle diverges from source). This is the guard that enforces "always rebuild and commit `dist/`."

### dependabot-dist.yml

On Dependabot PRs, rebuilds `dist/` and commits the result back to the PR branch, so a build-tool bump (e.g. esbuild) that changes the bundle doesn't fail `verify-dist`. **Do not hand-edit this workflow** -- it is already correct.

### publish-immutable-action.yml

Triggered when a GitHub release is published; uses `actions/publish-immutable-action` to publish the action to the GitHub Actions marketplace.

## Common Tasks for AI Agents

### Adding a new action input

1. Add the input in `action.yml` under `inputs:`.
2. Read it in `index.js` with `core.getInput('input-name')` (add `{ required: true }` only if truly required).
3. If it feeds the blueprint, thread it through the `options` object passed to `buildBlueprint()` and wire it into the blueprint in `lib.js` (add a section only when the value is present, following the existing "add only when set" pattern).
4. Add/extend a unit test in `test/index.test.js`.
5. Document the input in `README.md` (and here).
6. Run `npm run build` and commit the regenerated `dist/index.cjs`.

### Changing the blueprint defaults or shape

Edit `buildBlueprint()` (and its helpers `buildPrimaryAddon`, `dedupeAddons`, `mergeBlueprint`) in `lib.js`. Keep the "optional sections added only when present" invariant and the final `blueprint-json` deep-merge/dedup step. Update the corresponding `deepEqual` assertions in `test/index.test.js`.

### Adjusting the URL-length warning

The threshold is `MAX_SAFE_PREVIEW_URL` in `lib.js`; the predicate is `previewUrlExceedsLimit`; the warning text lives in `index.js` right after `buildPreviewUrl()`. The warning must stay advisory (`core.warning`, never `core.setFailed`).

### Testing locally

There is a real unit-test suite (unlike some sibling actions). To verify changes:

1. Edit `index.js` / `lib.js`.
2. `npm test` -- all tests in `test/index.test.js` must pass.
3. `npm run build` and confirm `git status` shows `dist/index.cjs` in sync (or commit the regenerated bundle).
4. For end-to-end verification, push a branch and open a test PR on a repo that uses this action; check the PR comment/description for the preview link.

### Publishing a new version

1. Ensure `dist/index.cjs` is up to date (`npm run build`) and committed.
2. Bump `package.json` `version` and commit.
3. Create a GitHub release with a semver tag (e.g. `v1.0.0`). `publish-immutable-action.yml` publishes to the marketplace automatically on release.
4. Move the major-version tag so consumers pinning `@v1` get the update:
   `git tag -fa v1 -m "Update v1 tag" && git push origin v1 --force`.

   (Existing tags in this repo: `v1`, `v1.0.0`.)
